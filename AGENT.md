## 3. 작업 절차 (에이전트 필독)

### 시작할 때
1. **`git log --oneline -5`와 `package.json` version부터 확인** — 이 저장소의
   소유자는 세션 사이에 직접 커밋 amend·버전 범프(`0.6.3` 같은 커밋)·브랜치
   머지·배포를 합니다. 마지막으로 알던 상태를 믿지 마세요.
2. 규칙·수치를 바꾸는 작업이면 `CLAUDE.md`의 해당 §를 먼저 읽으세요.

### 작업할 때 — 불변 원칙 (위반 금지)
- **초당 60번 변하는 것은 캔버스에, 가끔 변하는 것만 React에** (CLAUDE.md §12)
- **부가 기능(저장·소리·네트워크)이 게임을 죽이면 안 된다** — 전부 try-catch, 실패 시 조용히
- **에셋 파일 0개** — 그림·소리·아이콘은 코드로 생성
- **색은 `src/lib/constants.ts`에서만** — 하드코딩 색 금지
- 픽셀 아트는 **-4~+4 가상 격자 + scale** 문법 (docs/04), 캔버스 텍스트는 영어만
  (Press Start 2P에 한글 없음 — 한글 UI는 HTML + `font-pixel-ko`)
- update/draw 분리, 모든 이동은 ×dt, 배열 삭제는 역순 splice
- DB **쓰기는 RPC(`register_pet`·`submit_result`)로만** — 테이블 직접 쓰기 정책을 열지 말 것

### 끝낼 때 — 동기화 의무
- 규칙·수치가 바뀌었으면 → `CLAUDE.md` 해당 § 갱신
- 문법·구조가 크게 바뀌었으면 → `docs/` 해당 챕터 갱신
- 작업 절차·이슈·로드맵이 바뀌었으면 → 이 파일(AGENT.md) 갱신
- 커밋 메시지: `feat:`/`fix:`/`docs:` + 한글 요약 + 관련 § 표기 (기존 로그 참고)

## 4. 검증 루틴

```bash
npx tsc --noEmit          # 타입
npm run lint              # 린트
npm run build             # 프로덕션 빌드 (dev 서버 잠시 내리고)
npm run dev               # 3004에서 확인
```

- **그리기 코드 검증**: 브라우저 없이도 가능 — 가짜 ctx(fillRect + 변환행렬만
  구현)로 실제 draw 함수를 실행해 ASCII로 찍어 보는 기법을 씁니다.
  `npx tsc src/lib/{debris,mascot,constants}.ts --outDir <임시> --module commonjs`로
  컴파일한 뒤 스텁 ctx로 render. (과거 세션에서 스프라이트 전수 검증에 사용 —
  스크립트는 세션 scratchpad에 있었으므로 필요하면 다시 만들 것.)
- **리더보드 e2e**: `.env.local`을 source해서 curl로 REST/RPC를 직접 때려봅니다.
  테스트 데이터를 넣었으면 **반드시 정리 SQL을 사용자에게 전달** (anon 키로는
  삭제 불가 — 의도된 RLS).
- **서비스 워커·설치 프롬프트**: headless 검증 불가 — 배포 후 실기기 확인 필요.

## 5. 배포 루틴

1. `package.json`의 `version`을 올린다 — **이게 설치된 앱의 갱신 토스트 트리거**
   (`/sw.js?v={version}` 등록 URL이 바뀌어야 새 워커가 설치됨)
2. DB 변경이 있으면 **`supabase/schema.sql`을 먼저 SQL Editor에서 실행**
   (파일은 재실행-안전한 마이그레이션 스타일로 유지할 것) → 그다음 푸시·배포
3. 푸시·배포는 저장소 소유자가 직접 한다 — 에이전트는 커밋까지

## 6. 알려진 이슈 · 기술 부채

- **[UI] `/rank`·`/bag` 하단 버튼 밀림**: 목록이 길어지면 TAP TO START·HOME이
  화면 밖으로. "하단 고정 액션 바" 안을 만들다 사용자가 취소함 — 다른 접근
  (버튼 상단 배치, 목록 고정 높이 스크롤 등)을 상의 후 진행할 것.
- **[유지보수] 앱 아이콘 이중 정의**: `pwa-icon.tsx`는 `mascot.ts` 도트 좌표의
  복사본 — 마스코트 도트를 바꾸면 아이콘도 같이 바꿔야 한다 (파일 주석에 명시).
- **[정합성] 로컬 인벤토리 vs 서버 누적**: `sjs-inventory`(기기별)와
  `pets.total_eaten`(서버)은 별개 집계 — 의도된 설계지만 혼동 주의.
- **[명명] `package.json`의 name이 `jd-02`**: 역사적 흔적. 바꿔도 무방하나 우선순위 낮음.
- **[데이터] `scores` 테이블**: 현재 랭킹에 안 쓰이는 판 히스토리 — 크기가 계속
  자라므로 언젠가 보존 기간 정책 필요할 수 있음.

## 7. 향후 발전 로드맵 (구현 전 사용자와 상의)

**명세 잔여** (§16):
- 간단한 도전 과제 ("한 판에 별 3개" 등) — 로컬 도감과 결합하면 자연스러움

**확장 후보** (우선순위 논의 필요):
- 인벤토리 서버 동기화 / 도감·기록 "자랑 카드" 공유 (STELLAPET §10의 자랑 카드처럼)
- `scores` 히스토리 활용: 최근 판 그래프, 개인 통계 페이지
- 펫 이전(다른 기기로): 짧은 코드 발급 → localStorage 복원
- 효과음 볼륨 조절 (현재는 on/off만)
- 오프라인 중 기록 큐잉 → 온라인 복귀 시 일괄 제출 (지금은 조용히 유실)
- i18n: 영어 UI 모드 (스토리·도감의 영어판)
- 밸런스 튜닝: scores 데이터 기반으로 난이도 곡선·파워업 확률 재조정
- 자매작 STELLAPET(jd-03)과의 실제 연동 (펫 데이터 교환)

## 8. 데이터베이스 요약 (Supabase)

| 객체 | 역할 |
|---|---|
| `pets` | 펫 1행: 이름(유니크, lower), `total_eaten`(누적 랭킹), `best_score`+`best_at`(단판 랭킹) |
| `scores` | 판 히스토리 (랭킹 원천 아님) |
| RPC `register_pet(id, name)` | 이름 선점 — 중복이면 409, 멱등 |
| RPC `submit_result(id, score, eaten)` | 누적 가산 + best 갱신 + 히스토리 insert (원자적) |
| RLS | 읽기 공개 / 쓰기는 RPC만 (anon 직접 INSERT/UPDATE 불가) |

클라이언트는 SDK 없이 `src/lib/leaderboard.ts`에서 fetch로 직접 호출 (docs/07).
