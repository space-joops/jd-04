# 07. 서버와 데이터 — 온라인 랭킹은 어떻게 돌아가나

## 1. 큰 그림

```
브라우저(게임) ──fetch(HTTP)──▶ Supabase (PostgREST) ──▶ PostgreSQL(DB)
```

- **PostgreSQL**: 표(테이블)에 데이터를 쌓는 데이터베이스.
- **Supabase**: 그 DB를 인터넷에서 쓸 수 있게 REST API로 감싸 주는 서비스.
- 우리는 SDK(라이브러리) 없이 **fetch로 직접** 호출합니다 — 요청이 세 종류뿐이라,
  의존성을 늘리는 것보다 HTTP가 눈에 보이는 쪽이 배우기 좋아서요.

## 2. fetch — HTTP 요청의 기본기

```ts
// src/lib/leaderboard.ts — 누적 랭킹 읽기 (GET)
const res = await fetch(
  `${URL}/rest/v1/pets?select=id,name,total_eaten` +
    `&order=total_eaten.desc,updated_at.asc&limit=${limit}`,
  { headers: HEADERS },
);
if (!res.ok) return null;          // 200번대가 아니면 실패로 처리
const rows: unknown = await res.json();   // 응답 본문(JSON) 꺼내기
```

- **GET**(읽기)은 주소만으로, **POST**(쓰기)는 `method`와 `body`를 함께:

```ts
// 기록 제출 (POST) — RPC 함수 호출
const res = await fetch(`${URL}/rest/v1/rpc/submit_result`, {
  method: "POST",
  headers: HEADERS,                     // 인증 키 + "본문은 JSON이야"
  body: JSON.stringify({ p_id: pet.id, p_score: 10, p_eaten: 3 }),
});
```

- **상태 코드** 읽는 법: 200/204 성공, 400 잘못된 요청, 401 인증 실패,
  404 없음, 409 충돌(우리는 "이름 이미 있음"으로 사용).

## 3. PostgREST 쿼리 문법 — 주소로 SQL 쓰기

Supabase의 REST는 주소의 쿼리스트링이 곧 질의입니다:

| 조각 | 뜻 |
|---|---|
| `/rest/v1/pets` | pets 테이블에서 |
| `select=id,name,score:best_score` | 이 컬럼들만 (`score:best_score`는 **별명** — 응답 필드명이 score가 됨) |
| `best_score=gt.0` | best_score > 0 인 행만 (gt = greater than) |
| `order=best_score.desc,best_at.asc` | 정렬: 점수 내림차순, 동점이면 먼저 세운 쪽 |
| `limit=10` | 10개까지만 |

## 4. 환경 변수 — 비밀과 설정을 코드 밖에

```ts
const URL = process.env.NEXT_PUBLIC_SUPABASE_URL;
const KEY = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY;
export const leaderboardEnabled = Boolean(URL && KEY);
```

- `.env.local` 파일의 값이 빌드 때 주입됩니다 (git에는 안 올라감 — `.env.example` 참고).
- `NEXT_PUBLIC_` 접두사가 붙어야만 브라우저 번들에 들어가요.
- 값이 없으면? `leaderboardEnabled`가 false — **리더보드 UI가 통째로 숨고
  게임은 로컬 기록만으로 잘 돌아갑니다.** "부가 기능이 게임을 죽이면 안 된다".

anon key는 공개돼도 되는 키예요. 진짜 보안은 키가 아니라 DB 쪽 규칙(RLS)이 담당합니다.

## 5. SQL 첫걸음 — supabase/schema.sql 읽기

SQL은 "표에게 말 거는 언어"입니다. 우리 스키마의 핵심만:

```sql
create table if not exists public.pets (
  id uuid primary key,                 -- 펫의 고유 번호 (경쟁 키)
  name text not null
    check (char_length(name) between 1 and 10),   -- 규칙 위반 데이터는 거부
  total_eaten bigint not null default 0,          -- 통산 수거량 (누적 랭킹)
  best_score integer not null default 0           -- 단판 최고 (단판 랭킹)
);

create unique index pets_name_unique_idx on public.pets (lower(name));
-- lower() = 대소문자 무시 유니크: "Tom"과 "TOM"은 같은 이름
```

- `check (...)`: DB 스스로 지키는 규칙. 클라이언트 검증은 우회할 수 있지만
  이건 못 지나갑니다.
- `index`: 자주 쓰는 정렬/검색을 빠르게 하는 색인. "읽기 패턴 그대로" 만들어요.

## 6. RLS — 누가 무엇을 할 수 있나

```sql
alter table public.pets enable row level security;

create policy "pets_select_all" on public.pets
  for select using (true);    -- 읽기: 누구나
-- insert/update 정책 없음 = anon 키로는 직접 쓰기 전부 거부!
```

RLS(Row Level Security)는 "정책이 없으면 기본 거부"예요. 그래서 공개된
anon 키를 누가 훔쳐 가도 남의 기록을 지우거나 조작할 수 없습니다 —
읽기와, 아래 RPC 함수 호출만 가능해요.

## 7. RPC 함수 — 서버 안에서 원자적으로

"펫 누적치 더하기 + 단판 기록 남기기"를 클라이언트가 두 번의 요청으로 하면
중간에 끊기거나 경합이 생길 수 있어요. 그래서 **DB 안의 함수**로 묶습니다:

```sql
create or replace function public.submit_result(
  p_id uuid, p_score integer, p_eaten integer
) returns void
language plpgsql
security definer            -- ★ 관리자 권한으로 실행 (RLS를 지나간다)
as $$
begin
  if p_score not between 1 and 100000 then
    raise exception 'invalid score';     -- 함수가 직접 검증
  end if;

  update pets set
    total_eaten = total_eaten + p_eaten,             -- 누적 가산이 DB 안에서
    best_at = case when p_score > best_score
                   then now() else best_at end,      -- 기록 깰 때만 시각 갱신
    best_score = greatest(best_score, p_score)
  where id = p_id;
  if not found then raise exception 'unregistered pet'; end if;

  insert into scores (name, score, eaten, pet_id)
  select name, p_score, p_eaten, p_id from pets where id = p_id;
end;
$$;
grant execute on function public.submit_result(uuid, integer, integer) to anon;
```

- `security definer`: 함수는 관리자 권한 — 대신 **함수 안에서 철저히 검증**.
  anon에게는 이 함수의 "실행 권한만" 줍니다. 좁은 문 하나만 열어 두는 거예요.
- 호출은 REST로: `POST /rest/v1/rpc/submit_result` + JSON 본문(인자 이름 그대로).

`register_pet`(이름 선점)도 같은 구조인데 한 가지 트릭이 있어요 — 이름이
이미 있으면 유니크 인덱스 위반(SQL 에러 23505)이 나고, PostgREST가 이를
**HTTP 409**로 번역해 줍니다. 클라이언트는 상태 코드만 보면 되죠:

```ts
if (res.ok) return "ok";
if (res.status === 409) return "taken";   // "이미 있는 이름이야!"
return "error";
```

## 8. 실패 허용 설계 — null과 false의 미학

`src/lib/leaderboard.ts`의 모든 함수는 **절대 throw하지 않습니다**:

```ts
} catch {
  return null;   // 오프라인·CORS·타임아웃 — 어떤 실패든 조용히
}
```

호출하는 쪽(UI)은 null이면 목록을 숨기고, false면 "전송 실패" 한 줄을
보여줄 뿐 — 게임은 어떤 경우에도 멈추지 않아요. 네트워크가 없는 지하철에서도
게임이 되는 이유입니다.
