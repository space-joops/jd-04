# 04. 캔버스와 게임 루프 — 게임이 매끄럽게 움직이는 비밀

## 1. 캔버스가 뭔가요?

`<canvas>`는 "자바스크립트로 그림을 그릴 수 있는 도화지"입니다.
HTML 요소들과 달리, 그린 다음엔 그냥 픽셀 덩어리예요 — 그래서 매 프레임
전부 지우고 다시 그리는 게 기본입니다.

```tsx
const canvas = canvasRef.current;
const ctx = canvas?.getContext("2d");   // ctx = 붓. 모든 그리기는 ctx로.
```

기본 붓질 두 가지만 알면 우리 게임의 그림 대부분을 읽을 수 있어요:

```ts
ctx.fillStyle = "#7ee8b2";        // 붓에 물감 묻히기
ctx.fillRect(x, y, 너비, 높이);    // 사각형 칠하기 — 픽셀 아트의 전부!
```

## 2. 게임 루프 — requestAnimationFrame

애니메이션의 원리는 단순해요: **1초에 60번, 지우고 다시 그린다.**

```ts
// src/app/play/joops-game.tsx
let raf = 0;
let last = performance.now();
const frame = (now: number) => {
  const dt = Math.min(TUNE.maxDt, (now - last) / 1000);
  last = now;
  if (!paused) update(dt);   // ① 세상을 움직이고
  draw();                    // ② 그 결과를 그린다
  raf = requestAnimationFrame(frame);   // ③ 다음 프레임 예약
};
raf = requestAnimationFrame(frame);
```

- `requestAnimationFrame(f)` — "다음 화면 갱신 타이밍에 f를 불러 줘".
  브라우저의 화면 주사율(보통 60Hz)에 맞춰 부드럽게 돌아갑니다.
- 자기 자신을 다시 예약하는 구조라 무한 루프처럼 돌아요.
- 페이지를 떠날 땐 `cancelAnimationFrame(raf)`로 꼭 멈춥니다 (03챕터의 정리 함수).

## 3. dt — 프레임레이트에서 독립하기

`dt`(delta time)는 **지난 프레임에서 지금까지 흐른 시간(초)**입니다.
모든 움직임에 dt를 곱하는 게 이 게임의 철칙이에요:

```ts
j.y += j.vy * dt;   // 속도(px/초) × 시간(초) = 이동 거리(px)
```

왜냐면 — 120Hz 폰에서는 frame이 두 배 자주 불립니다. dt를 안 곱하면
게임이 두 배 빨라져요! dt를 곱하면 "1초에 vy픽셀"이라는 약속이 어떤
기기에서든 지켜집니다.

```ts
const dt = Math.min(TUNE.maxDt, (now - last) / 1000);
```

`Math.min(0.05, ...)`로 dt에 **상한**을 두는 것도 중요해요. 백그라운드 탭에
갔다 오면 dt가 수십 초가 되는데, 그대로 곱하면 낙하물이 화면을 뚫고
순간이동합니다(터널링). 최대 0.05초로 잘라서 방지해요.

## 4. update / draw 분리 — 이 게임의 헌법

```ts
const update = (dt: number) => { /* 상태만 바꾼다. 캔버스는 안 건드림 */ };
const draw = () => { /* 읽어서 그리기만 한다. 상태는 안 바꿈 */ };
```

왜 나눌까요? 섞이면 "그리는 순서를 바꿨더니 게임 규칙이 달라지는" 지옥이
열립니다. 물리는 물리대로, 그림은 그림대로 — 디버깅이 반은 쉬워져요.
(일시정지도 이 분리 덕분에 한 줄입니다: update만 건너뛰고 draw는 계속!)

## 5. 좌표 변환 — translate / rotate / scale / save / restore

낙하물마다 위치·회전이 다른데, 그림 코드는 어떻게 하나로 쓸까요?
**종이를 움직이는** 발상입니다:

```ts
// src/lib/debris.ts — drawJunk
ctx.save();                          // 지금 좌표계를 스냅숏
ctx.translate(j.x, j.y);             // 원점을 낙하물 위치로 이동
ctx.rotate(j.rot);                   // 종이를 회전
ctx.scale(scale * (s / 4), ...);     // 종이를 확대
// 이제 "원점 근처에 그리면" 알아서 그 위치에·그 각도로·그 크기로 찍힌다
ctx.fillRect(-2, -3, 4, 6);
ctx.restore();                       // 스냅숏 복원 — 다음 그림에 영향 없게
```

`save()`/`restore()`는 항상 짝으로! 안 그러면 변환이 누적돼 화면이 뒤틀립니다.

**globalAlpha(투명도)는 대입이 아니라 곱하기**로 쓰는 것도 같은 이유예요:

```ts
ctx.globalAlpha *= 0.5;   // 부모가 반투명이면 그 위에 다시 반투명 — 자연스럽게 중첩
```

## 6. 픽셀 아트 기법 — "가상 픽셀 격자"

마스코트와 낙하물은 전부 이 방식으로 그립니다:

```ts
// src/lib/mascot.ts
const scale = r / 4;        // 반지름 4 = 가상 픽셀 1칸
ctx.scale(scale, scale);
ctx.fillRect(-2, -4, 4, 1); // -4~+4 범위의 "칸" 좌표로 도트를 찍는다
```

도형을 실제 픽셀이 아니라 **-4~+4의 가상 격자**에 그리고, scale로 확대해요.
몸이 커져도(성장 시스템) 도트가 같이 굵어져서 실루엣이 유지됩니다.

배경 별처럼 변환 없이 직접 찍을 때는 좌표를 정수로 내립니다:

```ts
ctx.fillRect(Math.floor(x), Math.floor(y), size, size);
// 소수 좌표는 안티앨리어싱으로 도트가 뭉개진다
```

## 7. 화면 흔들림 — 좌표계 전체를 흔든다

```ts
// draw() 맨 앞
if (shake > 0) {
  const power = (shake / TUNE.shakeTime) * TUNE.shakeAmp;
  ctx.translate((Math.random() * 2 - 1) * power, (Math.random() * 2 - 1) * power);
}
```

오브젝트를 하나하나 흔드는 게 아니라 **종이(좌표계)를 통째로** 흔들어요.
`shake`가 줄어들수록 진폭도 줄어 자연스럽게 잦아듭니다.

## 8. DPR — 레티나에서 또렷하게

```ts
// src/lib/canvas.ts — fitCanvas
const dpr = Math.min(2, window.devicePixelRatio || 1);
canvas.width = w * dpr;      // 실제 픽셀은 2배로
canvas.height = h * dpr;
ctx.scale(dpr, dpr);         // 좌표계는 CSS 픽셀 그대로
```

고해상도 화면은 CSS 1픽셀에 실제 픽셀이 2~3개예요. 캔버스를 그만큼 크게
만들고 scale로 좌표를 맞추면 흐릿함이 사라집니다. 상한 2는 화질과 성능의 타협.

## 9. 배열 순회 중 삭제 — 역순 + splice

```ts
for (let i = junks.length - 1; i >= 0; i--) {
  const j = junks[i];
  // ...
  if (j.y > h + 70) junks.splice(i, 1);   // i번째 제거
}
```

앞에서부터 돌며 지우면 뒤 요소들이 한 칸씩 당겨져 **다음 요소를 건너뛰는
버그**가 생깁니다. 뒤에서부터 돌면 지워도 앞쪽 인덱스가 안 흔들려요.

## 10. 객체 + 순수 함수 — 클래스 없는 엔티티

낙하물은 클래스가 아니라 "평범한 객체 + 그걸 다루는 함수 3형제"입니다:

```ts
makeJunk(kind, w, speedScale)  // 데이터 만들기
stepJunk(j, dt)                // 물리 한 걸음 (상태만 변경)
drawJunk(ctx, j, scale)        // 그리기 (읽기만)
```

상속 계층 없이도 충분히 정리되고, 함수마다 역할이 하나라 테스트하기 쉬워요.
`render-test`(가짜 ctx로 ASCII 렌더링)로 그림을 검증할 수 있는 것도 이 구조 덕분입니다.
